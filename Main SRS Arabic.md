## 1. Introduction - المقدمة

### 1.1 Purpos - الغرض
توفر هذه الوثيقة مواصفات متطلبات برمجيات شاملة (SRS) لنظام إدارة متجر إلكتروني متكامل مصمم لأتمتة جميع العمليات التجارية لمتجر بيع مفرق كبير في الجمهورية العربية السورية. يهدف النظام إلى رقمنة سير العمل التجاري بالكامل من عرض المنتجات إلى إدارة المستودعات، تنفيذ الطلبات، والعمليات المالية.

### 1.2 Scope - النطاق
يشمل النظام تطبيقين رئيسيين:
- **تطبيق العملاء (Mobile/Web)**: للمستخدمين النهائيين لتصفح والبحث وشراء المنتجات
- **لوحة الإدارة (Web Application)**: لمسؤولي النظام والموظفين لإدارة جميع جوانب العمل

### 1.3 Business Context - السياق التجاري
النظام مصمم لمتجر بيع مفرق كبير يتميز بـ:
- مبيعات يومية تتجاوز 1000 معاملة
- فئات منتجات متنوعة (إلكترونيات، أغذية، مواد منزلية، منتجات طبية)
- مخزون مستودع كبير
- توصيل إلى محافظات سورية متعددة
- حاجة للأتمتة الكاملة وتقليل الزيارات للمتجر الفعلي

## 2. Overall Description الوصف العام

### 2.1 System Description وصف النظام
نظام المتجر الإلكتروني هو منصة شاملة تقوم بأتمتة:
- إدارة كتالوج المنتجات
- Authentication & authorization للعملاء
- عربة التسوق ومعالجة الطلبات
- تتبع وإدارة المخزون
- سير عمل تنفيذ الطلبات (workflow)
- معالجة المعاملات المالية
- إدارة التوصيل والشحن
- إدارة الموظفين والأدوار
- التقارير والتحليلات

### 2.2 Product Functions - وظائف المنتج

يوفر هذا القسم ملخصاً احترافياً للوظائف الرئيسية التي يجب أن يؤديها نظام إدارة المتجر الإلكتروني. يتم توفير المواصفات التفصيلية لكل وظيفة في القسم 3 (المتطلبات الوظيفية). يتكون النظام من تطبيقين رئيسيين بمتطلبات وظيفية مميزة كما هو موضح أدناه.

#### 2.2.1 Customer Application Functions - وظائف تطبيق العملاء

**اكتشاف المنتجات والتصفح**
- عرض المنتجات حسب الفئات والعلامات التجارية مع إمكانيات البحث والتصفية والفرز
- صفحة رئيسية قابلة للتخصيص مع المنتجات المميزة، المنتجات الجديدة، والعروض الترويجية
- تنوعات المنتجات (الحجم، اللون، إلخ) مع تسعير ومخزون فردي (variations)

**مصادقة العملاء (Authentication)**
- مصادقة موحدة عبر رقم الهاتف و OTP
- توجيه المستخدمين الحاليين إلى الشاشة الرئيسية؛ المستخدمين الجدد إلى إعداد الحساب
- إدارة الجلسات (sessions)

**إدارة عربة التسوق والطلبات**
- إضافة المنتجات إلى العربة، تعديل الكميات، وإزالة العناصر
- تقديم الطلبات مع التحقق من OTP، اختيار عنوان التوصيل، وحساب الرسوم بناءً على المحافظة
- عرض سجل الطلبات، تتبع حالة الطلب في الوقت الفعلي، وإعادة طلب الطلبات السابقة
- كشف تلقائي للمحافظة لحساب رسوم التوصيل

#### 2.2.2 Admin Panel Functions - وظائف لوحة الإدارة

**إدارة المحتوى**
- إدارة الفئات، العلامات التجارية، والمنتجات مع تنظيم هرمي
- تخصيص محتوى الصفحة الرئيسية وجدولة العروض الترويجية

**إدارة العملاء والطلبات**
- إنشاء وإدارة حسابات العملاء مع توليد كلمة مرور تلقائية
- عرض وتحديث والبحث في الطلبات عبر مراحل سير العمل المحددة
- إكمال تلقائي للطلب عند تسجيل الدفع

**إدارة المخزون**
- تتبع المخزون في الوقت الفعلي مع تحديثات تلقائية عند إتمام الطلب
- تنبيهات المخزون المنخفض وتعديلات المخزون اليدوية
- منع البيع الزائد من خلال فحص التوفر (check availability)

**التوصيل والتقارير**
- إدارة محافظات التوصيل وتحديد الرسوم لكل محافظة
- إنشاء تقارير المبيعات والتشغيل مع إمكانيات التصدير (PDF, Excel/CSV)
- التحكم في الوصول القائم على الأدوار (RBAC) مع أدوار مخصصة وصلاحيات دقيقة

#### 2.2.3 Functional Relationships - العلاقات الوظيفية

المجموعات الوظيفية الرئيسية مترابطة:
 - اكتشاف المنتجات (Product Discovery) يغذي إدارة عربة التسوق (Shopping Cart Management) ومعالجة الطلبات (Order Processing).
 - مصادقة العملاء (Customer Authentication) مطلوبة لعمليات العربة والطلبات. معالجة الطلبات (Order Processing) تحفز تحديثات إدارة المخزون (Inventory Management).
 - إدارة المحتوى (Content Management) تؤثر على اكتشاف المنتجات (Product Discovery).
 - التقارير (Reporting) تجمع البيانات من إدارة الطلبات والمخزون والعملاء.
 - التحكم في الوصول القائم على الأدوار (RBAC) يتحكم في الوصول لجميع وظائف الإدارة (Admin Functions).

سيوضح مخطط الفئات (Class Diagram) هذه العلاقات، موضحاً تدفق البيانات من إجراءات العملاء إلى معالجة الإدارة والتحديثات في الوقت الفعلي للعملاء.

### 2.3 System Architecture - معمارية النظام

#### 2.3.1 Architecture Overview - نظرة عامة على المعمارية
يتبع النظام معمارية ثلاثية الطبقات (three-tier architecture):
- **طبقة العرض (Presentation Layer)**: تطبيق العملاء ولوحة الإدارة
- **طبقة منطق الأعمال (Business Logic Layer)**: server التطبيق مع قواعد الأعمال
- **طبقة البيانات (Data Layer)**: قاعدة البيانات للتخزين الدائم

#### 2.3.2 Technology Stack - حزمة التقنيات (الموصى بها)
- **Frontend**: React/Vue.js أو Flutter (للموبايل)
- **Backend**: Node.js/Laravel
- **Database**: PostgreSQL/MySQL
- **Caching**: Redis
- **File Storage**: AWS or local storage
- **SMS Gateway**: التكامل (integration) مع مزودي الاتصالات السوريين (سيريتل، MTN)

#### 2.3.3 Integration Points - نقاط التكامل
- SMS Gateway لـ OTP والإشعارات
- Payment Gateway مثل Syriatel-Cash، Sham-Cash، إلخ..
- خدمات GPS/تحديد الموقع الجغرافي
- خدمة البريد الإلكتروني (للإشعارات)

### 2.4 Assumptions And Constraints - الافتراضات والقيود

#### 2.4.1 Assumptions - الافتراضات
- العملاء لديهم إمكانية الوصول إلى الهواتف الذكية أو أجهزة الكمبيوتر
- الاتصال بالإنترنت متاح
- خدمات SMS موثوقة ومتاحة
- خدمات GPS/تحديد الموقع الجغرافي متاحة
- ستتم معالجة الدفع (الدفع الفيزيائي/ Sham-Cash/ Syriatel-Cash)

#### 2.4.2 Constraints القيود
- يجب أن يدعم التوصيل للمحافظات السورية
- يجب أن يتكامل مع مزودي الاتصالات السوريين (سيريتل، MTN)
- يجب أن يتبع القوانين المحلية
- محدود بحزمة التقنيات المتاحة (technology stack)

## 3. Functional Requirements - المتطلبات الوظيفية

### 3.1 Customer Application Requirements - متطلبات تطبيق العملاء

#### 3.1.1 Product Management - إدارة المنتجات

**FR-CA-001: Product Display - عرض المنتجات**
- **الوصف**: يجب أن يعرض النظام المنتجات منظمة حسب الفئات والعلامات التجارية
- **الأولوية**: High
- **التفاصيل**:
  - يجب تنظيم المنتجات في فئات واضحة وهرمية
  - يجب ربط المنتجات بالعلامات التجارية
  - يجب أن تتضمن قوائم المنتجات: الاسم، الصورة، السعر، حالة التوفر، التقييم
  - يجب أن تدعم المنتجات صوراً متعددة لكل عنصر
  - يجب عرض كمية المخزون (الكمية المتاحة) في الوقت الفعلي وتحديثها تلقائياً مع تغيرات المخزون

**FR-CA-002: Dynamic Homepage الصفحة الرئيسية الديناميكية**
- **الوصف**: يجب أن يوفر النظام صفحة رئيسية قابلة للتخصيص تُدار من لوحة الإدارة
- **الأولوية**: High
- **التفاصيل**:
  - يجب أن تعرض الصفحة الرئيسية المنتجات المميزة
  - يجب أن تعرض الصفحة الرئيسية المنتجات الجديدة
  - يجب أن تعرض الصفحة الرئيسية العروض الترويجية
  - يجب أن تعرض الصفحة الرئيسية المنتجات الخاصة/المميزة
  - يجب أن يكون المحتوى قابلاً للترتيب الديناميكي من قبل المسؤولين

**FR-CA-003: Advanced Product Filtering تصفية متقدمة للمنتجات**
- **الوصف**: يجب أن يوفر النظام إمكانيات تصفية (filtering) متقدمة لبحث المنتجات
- **الأولوية**: High
- **التفاصيل**:
  - التصفية حسب نطاق السعر (الحد الأدنى والأقصى)
  - التصفية حسب الفئة
  - التصفية حسب العلامة التجارية
  - التصفية حسب تقييم العملاء
  - دعم مرشحات متعددة في وقت واحد
  - توفير خيارات الفرز (sorting) (السعر، التقييم، تاريخ الإضافة، الشعبية)

**FR-CA-004: Product Variations تنوعات المنتجات**
- **الوصف**: يجب أن يدعم النظام المنتجات ذات التنوعات المتعددة (مثل الحجم، اللون)
- **الأولوية**: High
- **التفاصيل**:
  - قد تحتوي المنتجات على سمات متعددة (الحجم، اللون، المادة، إلخ)
  - يجب أن يكون لكل تنوع SKU خاص به، السعر، والمخزون
  - يجب أن تكون التنوعات قابلة للاختيار في صفحات تفاصيل المنتج
  - يجب تتبع المخزون لكل تنوع

**FR-CA-005: Product Search بحث المنتجات**
- **الأولوية**: High
- **الوصف**: يجب أن يوفر النظام وظيفة بحث مع الإكمال التلقائي والاقتراحات

#### 3.1.2 Authentication and Authorization - مصادقة وتفويض العملاء

**FR-CA-005: User Authentication مصادقة المستخدم**
- **الأولوية**: Critical
- **الوصف**: يجب مصادقة جميع العملاء قبل استخدام التطبيق
- **المتطلبات**:
  - آلية تسجيل دخول آمنة
  - تشفير كلمة المرور
  - إدارة الجلسات (sessions)

**FR-CA-006: First-Time Password Activation تفعيل كلمة المرور للمرة الأولى**
- **الوصف**: يجب على العملاء المضافين من قبل المسؤول تفعيل كلمة المرور عند أول تسجيل دخول
- **الأولوية**: High
- **التفاصيل**:
  - إذا تم إنشاء الحساب من قبل المسؤول، يتم إرسال كلمة المرور عبر SMS
  - أول تسجيل دخول يتطلب تغيير كلمة المرور
  - فرض تغيير كلمة المرور قبل الوصول إلى الميزات الأخرى
  - كلمة المرور تلبي متطلبات الأمان (8 أحرف على الأقل، مزيج من الحروف والأرقام)

**FR-CA-007: Order Verification التحقق من الطلب (OTP)**
- **الوصف**: يجب التحقق من كل طلب عبر OTP قبل المعالجة
- **الأولوية**: High
- **التفاصيل**:
  - يُرسل OTP إلى رقم الهاتف المسجل
  - OTP صالح لفترة محدودة (مثل 5 دقائق)
  - لا تتم معالجة الطلب حتى التحقق من OTP
  - الغرض: التحقق من أن رقم الهاتف لا يزال نشطاً ولم يعاد تعيينه من قبل مزود الاتصالات

#### 3.1.3 Shopping Cart Management - إدارة عربة التسوق

**FR-CA-008**: ميزة عربة التسوق
  - **الوصف**: يجب أن يسمح النظام للعملاء بإضافة المنتجات إلى عربة التسوق مع القدرة على تعديل الكميات أو إزالة العناصر قبل الشراء
  - **الأولوية**: High
  - **التفاصيل**:
    - إضافة المنتجات إلى العربة
    - تحديث الكميات
    - إزالة العناصر
    - حفظ العربة لوقت لاحق
    - العربة موجودة عبر الجلسات المتعددة (sessions)

**FR-CA-009**: صفحة مراجعة العربة
  - **الوصف**: يجب أن يوفر النظام صفحة عربة تعرض جميع المنتجات المضافة مع الكميات والأسعار والمجموع الكلي
  - **الأولوية**: High
  - **التفاصيل**: 
    - حساب الأسعار في الوقت الفعلي
    - عرض واضح لجميع الرسوم بما في ذلك التوصيل

#### 3.1.4 Order Management - إدارة الطلبات

**FR-CA-010: Order Placement تقديم الطلب**
- **الوصف**: يجب أن يتمكن العملاء من تقديم الطلبات من عربة التسوق
- **الأولوية**: Critical
- **التفاصيل**:
  - مراجعة ملخص الطلب قبل التأكيد
  - تأكيد عنوان التوصيل
  - اختيار محافظة التوصيل (كشف تلقائي من الموقع)
  - تطبيق رسوم التوصيل بناءً على المحافظة
  - إضافة معلومات الدفع (بحالة الدفع الإلكتروني)
  - التحقق من الطلب عبر OTP
  - استلام إشعار عند تأكيد الطلب

**FR-CA-014: Order History سجل الطلبات**
- **الوصف**: يجب أن يتمكن العملاء من عرض سجل الطلبات
- **الأولوية**: High
- **التفاصيل**:
  - سرد جميع الطلبات السابقة
  - عرض تفاصيل الطلب: المنتجات، الكميات، الأسعار، التواريخ
  - عرض حالة الطلب الحالية
  - تصفية الطلبات حسب الحالة
  - البحث في الطلبات حسب نطاق التاريخ أو رقم الطلب

**FR-CA-015: Order Status Tracking تتبع حالة الطلب**
- **الوصف**: يجب أن يتمكن العملاء من تتبع حالة الطلب في الوقت الفعلي
- **الأولوية**: High
- **التفاصيل**:
  - عرض حالة الطلب الحالية: قيد الانتظار، قيد التحضير، في طريق التوصيل، تم التوصيل، مكتمل
  -Pending, Preparing, Out for delviery, Delivered, Completed
  - إظهار سجل الحالة مع التفاصيل الزمنية
  - استلام إشعارات عند تغيير الحالة

**FR-CA-016: Reorder Functionality ميزة إعادة الطلب**
- **الوصف**: يجب أن يتمكن العملاء من إعادة طلب الطلبات السابقة بنقرة واحدة
- **الأولوية**: Medium
- **التفاصيل**:
  - زر "إعادة الطلب" على واجهة الطلبات السابقة
  - نسخ جميع المنتجات من الطلب السابق إلى العربة الحالية
  - السماح بالتعديل قبل التأكيد
  - التحقق من توفر المنتج الحالي قبل الإضافة إلى العربة

#### 3.1.5 Location and Delivery - الموقع والتوصيل

**FR-CA-017: Automatic Location Detection كشف الموقع التلقائي**
- **الوصف**: يجب أن يكتشف النظام محافظة العميل تلقائياً
- **الأولوية**: High
- **التفاصيل**:
  - استخدام GPS الجهاز
  - اختيار المحافظة تلقائياً في نموذج التوصيل
  - السماح بالتعديل اليدوي إذا لزم الأمر
  - حساب رسوم التوصيل بناءً على المحافظة

### 3.2 Admin Panel Requirements - متطلبات لوحة الإدارة

#### 3.2.1 Content Management - إدارة المحتوى

**FR-AD-001: Category Management إدارة الفئات**
- **الوصف**: يجب أن يتمكن المسؤولون من إدارة فئات المنتجات
- **الأولوية**: High
- **التفاصيل**:
  - إنشاء وتعديل وحذف الفئات
  - تنظيم الفئات بشكل هرمي (أب/ابن)
  - إضافة صور الفئات
  - تحديد ترتيب عرض الفئات
  - تفعيل/تعطيل الفئات (activate/deactivate)

**FR-AD-002: Brand Management إدارة العلامات التجارية**
- **الوصف**: يجب أن يتمكن المسؤولون من إدارة العلامات التجارية للمنتجات
- **الأولوية**: High
- **التفاصيل**:
  - إنشاء وتعديل وحذف العلامات التجارية
  - رفع شعارات (logos) العلامات التجارية
  - ربط العلامات التجارية بالفئات
  - تحديد ترتيب عرض العلامات التجارية

**FR-AD-003: Homepage Content Management إدارة محتوى الصفحة الرئيسية**
- **الوصف**: يجب أن يتمكن المسؤولون من تخصيص محتوى الصفحة الرئيسية
- **الأولوية**: High
- **التفاصيل**:
  - اختيار المنتجات المميزة
  - اختيار تلقائي للمنتجات الجديدة
  - إنشاء وإدارة اللافتات الترويجية (promotional banners)
  - تحديد ترتيب عرض أقسام الصفحة الرئيسية
  - جدولة المحتوى الترويجي (تواريخ البدء/الانتهاء)

**FR-AD-004: Product Management إدارة المنتجات**
- **الوصف**: يجب أن يتمكن المسؤولون من إدارة المنتجات
- **الأولوية**: High
- **التفاصيل**:
  - إنشاء وتعديل وحذف المنتجات
  - رفع صور متعددة للمنتجات
  - تحديد المنتجات وتنوعاتها (variants)
  - تعيين التسعير (السعر الأساسي، سعر البيع) للمنتج أو لكل تنوع
  - إضافة المنتجات للفئات والعلامات التجارية
  - تعيين كمية المنتج المتوفرة وتحديثها تلقائياً عند تغيرات المخزون
  - إدارة أوصاف المنتجات والمواصفات (descriptions and specifications)

**FR-AD-004A: Bulk Product Import استيراد المنتجات بشكل جماعي**
- **الوصف**: يجب أن يتمكن المسؤولون من استيراد المنتجات والتنوعات بشكل جماعي من Excel sheets
- **الأولوية**: Critical
- **التفاصيل**:
  - استيراد المنتجات مع جميع التفاصيل (الاسم، الوصف، الفئة، العلامة التجارية، السعر، الصور، إلخ) من ورقة Excel
  - استيراد تنوعات المنتجات (الحجم، اللون، المادة، إلخ) مع SKU فردي، السعر، والمخزون لكل تنوع
  - دعم ربط أعمدة Excel بحقول المنتجات
  - التحقق من صحة البيانات المستوردة (الحقول المطلوبة، أنواع البيانات، التحقق من التنسيق)
  - التعامل مع المنتجات المكررة (تحديث الموجودة أو تخطيها)
  - معاينة البيانات المستوردة قبل الاستيراد النهائي
  - الإبلاغ عن أخطاء الاستيراد وإحصائيات النجاح
  - دعم التراجع عن عمليات الاستيراد الفاشلة
  - يتحقق النظام من تنسيق بيانات Excel ويبلغ عن أي أخطاء استيراد

#### 3.2.2 Customer Management - إدارة العملاء

**FR-AD-005: Customer Account Creation إنشاء حساب العميل**
- **الوصف**: يجب أن يتمكن المسؤولون من إنشاء حسابات العملاء يدوياً أو استيراد العملاء الحاليين بشكل جماعي
- **الأولوية**: Critical
- **التفاصيل**:
  - **الإنشاء اليدوي**:
    - إضافة معلومات العميل: الاسم، رقم الهاتف، البريد الإلكتروني، العنوان
    - توليد كلمة مرور مؤقتة
    - إرسال كلمة المرور عبر SMS للعميل
    - تعيين حالة الحساب (نشط/غير نشط)
    - إضافة ملاحظات حول العميل
  - **الاستيراد الجماعي (للعملاء الحاليين للمتجر)**:
    - استيراد تفاصيل العميل من ورقة Excel (الاسم، رقم الهاتف، البريد الإلكتروني، العنوان)
    - يكمل النظام تلقائياً سير عمل إنشاء الحساب لكل عميل مستورد:
      - إضافة معلومات العميل من بيانات Excel
      - توليد كلمة مرور مؤقتة لكل عميل
      - إرسال كلمة المرور عبر SMS لرقم هاتف كل عميل
    - يتم تعيين جميع حسابات العملاء المستوردة تلقائياً على حالة **معطلة**
    - تبقى الحسابات معطلة حتى يسجل العميل الدخول ويغير كلمة المرور المؤقتة (انظر FR-CA-006)
    - يتحقق النظام من تنسيق بيانات Excel ويبلغ عن أي أخطاء استيراد

**FR-AD-006: Customer Account Management إدارة حساب العميل**
- **الوصف**: يجب أن يتمكن المسؤولون من إدارة حسابات العملاء الموجودة
- **الأولوية**: High
- **التفاصيل**:
  - عرض قائمة العملاء مع البحث والمرشحات (search and filtering)
  - عرض سجل طلبات العميل
  - تعطيل/إعادة تفعيل الحسابات
  - إعادة تعيين كلمات مرور العملاء

#### 3.2.3 Order Management إدارة الطلبات

**FR-AD-007: Order Status Management إدارة حالة الطلب**
- **الوصف**: يجب أن يتمكن المسؤولون من تحديث حالة الطلب
- **الأولوية**: High
- **التفاصيل**:
  - عرض جميع الطلبات مع الفلاتر (الحالة، التاريخ، العميل)
  - تحديث حالة الطلب عبر سير العمل:
    1. **قيد الانتظار (Pending)**: تم استلام الطلب، في انتظار المعالجة
    2. **قيد التحضير (Preparing)**: يتم تحضير الطلب في المستودع
    3. **في طريق التوصيل (Out for Delivery)**: تم إرسال الطلب للتوصيل
    4. **تم التوصيل (Delivered)**: تم توصيل الطلب للعميل
    5. **مكتمل (Completed)**: اكتمل الطلب (تلقائياً بعد الدفع)
  - لا يمكن تغيير حالة الطلبات المكتملة أو الملغاة

**FR-AD-008: Automatic Order Completion إكمال الطلب التلقائي**
- **الوصف**: يجب أن تتغير حالة الطلب تلقائياً إلى "مكتمل/ completed" بعد تسجيل الدفع
- **الأولوية**: High
- **التفاصيل**:
  - عندما يتم تسجيل الدفع مع رقم الطلب والمبلغ المطابق
  - يحدث النظام تلقائياً حالة الطلب إلى "مكتمل"
  - لا يلزم التدخل اليدوي
  - يجب أن يطابق الدفع الطلب تماماً (المبلغ ورقم الطلب)

**FR-AD-009: Order Vieweing And Search عرض الطلبات والبحث**
- **الوصف**: يجب أن يتمكن المسؤولون من عرض الطلبات والبحث فيها
- **الأولوية**: High
- **التفاصيل**:
  - عرض تفاصيل الطلب: العميل، المنتجات، الكميات، الأسعار، التواريخ
  - التصفية حسب الحالة، نطاق التاريخ، العميل، المحافظة
  - البحث برقم الطلب
  - تصدير بيانات الطلب

#### 3.2.4 Inventory Management - إدارة المخزون

**FR-AD-010: Inventory Tracking تتبع المخزون**
- **الوصف**: يجب أن يتتبع النظام مستويات المخزون لجميع المنتجات
- **الأولوية**: High
- **التفاصيل**:
  - تتبع المخزون في الوقت الفعلي لكل منتج وvariant
  - عرض مستويات المخزون الحالية
  - تتبع حركات المخزون (دخل/خرج)
  - دعم مستودعات متعددة

**FR-AD-011: Automatic Inventory Updates تحديثات المخزون التلقائية**
- **الوصف**: يجب أن يتحدث المخزون تلقائياً عند تقديم الطلبات وإلغائها
- **الأولوية**: High
- **التفاصيل**:
  - عند تأكيد الطلب، يقلل النظام تلقائياً كمية المنتج
  - تحديث المخزون لكل variant منتج فوراً عند تقديم الطلب
  - إذا تم إلغاء الطلب، يستعيد النظام تلقائياً المخزون (زيادة المخزون مرة أخرى)
  - هذا يمنع البيع الزائد عند تقديم الطلب
  - التحقق من التوفر قبل تأكيد الطلب لمنع تقديم طلبات لعناصر غير متوفرة (availability check)

**FR-AD-012: Low Stock Alerts تنبيهات المخزون المنخفض**
- **الوصف**: يجب أن ينبه النظام المسؤولين عندما ينخفض المخزون عن العتبة -threshold-
- **الأولوية**: High
- **التفاصيل**:
  - عتبة قابلة للتحديد لكل منتج
  - التنبيه عبر إشعارات على لوحة التحكم
  - التنبيه عبر البريد الإلكتروني/SMS (قابل للتحديد)
  - قائمة بالعناصر منخفضة المخزون في لوحة المعلومات
  - التنبيه عندما ينتهي مخزون المنتج

**FR-AD-013: Inventory Management Interface واجهة إدارة المخزون**
- **الوصف**: يجب أن يتمكن المسؤولون من تعديل المخزون يدوياً
- **الأولوية**: Medium
- **التفاصيل**:
  - إضافة/إزالة المخزون يدوياً
  - تسجيل تعديلات المخزون مع السبب
  - عرض سجل المخزون
  - تحديثات المخزون الجماعية (via excel sheet)

#### 3.2.7 Employee and Permission Management - إدارة الموظفين والصلاحيات (RBAC)

**FR-AD-019: Role Management إدارة الأدوار**

- **الأولوية**: High

  - إنشاء أدوار مخصصة (وليس أدوار ثابتة) -dynamic NOT static-

**FR-AD-020: Permission Management إدارة الصلاحيات**

  - تعطيل/إعادة تفعيل حسابات الموظفين
  - عرض سجلات نشاط الموظفين (activity logs)

**FR-AD-022: Permissioin Granularity دقة الصلاحيات**

- **الأولوية**: High

  - أمثلة الصلاحيات: "عرض المنتجات"، "تعديل المنتجات"، "حذف المنتجات"، "عرض الطلبات"، "تحديث حالة الطلب"، "عرض التقارير"، إلخ
  - يجب أن تغطي الصلاحيات جميع العمليات الرئيسية
  - دعم صلاحيات القراءة فقط وصلاحيات القراءة والكتابة

## 4. Non-Functional Requirements - المتطلبات غير الوظيفية

### 4.1 Performance Requirements - متطلبات الأداء

**NFR-001: Response Time وقت الاستجابة**
- وقت تحميل الصفحة: < 2 ثانية
- نتائج البحث: < 1 ثانية
- معالجة الطلب: < 5 ثواني
- إنشاء التقرير: < 10 ثواني للتقارير القياسية

**NFR-002: Scalability قابلية التوسع**
- دعم ما لا يقل عن 1000 عملية يومية
- دعم المستخدمين المتزامنين: 500+ عميل، 50+ مستخدم إداري بوقت واحد
- يجب أن تتعامل قاعدة البيانات مع 100,000+ منتج
- يجب أن يتوسع النظام أفقياً (horiznotally scaling)

**NFR-003: Availability التوفر**
- وقت تشغيل النظام: 99.5% (حوالي 3.6 ساعة توقف شهرياً)
- نوافذ الصيانة المجدولة (scheduled maintenance windows)

### 4.2 Security Requirements - متطلبات الأمان

**NFR-004: Data Security أمان البيانات**
- جميع البيانات المنقولة عبر HTTPS
- كلمات المرور المخزنة باستخدام secure hashing
- البيانات الحساسة مشفرة أثناء التخزين
- منع SQL Injection

**NFR-005: Authentication Security أمان المصادقة**
- متطلبات كلمة مرور قوية
- مصادقة قائمة على الtokens (Token-based)
- انتهاء مهلة الجلسة بعد عدم النشاط (sessions)
- الحماية ضد هجمات brute force
- انتهاء مدة OTP والاستخدام لمرة واحدة

**NFR-006: Authorization أمان التفويض**
- التحكم في الوصول القائم على الأدوار مطبق على مستوى API
- مبدأ تقليل الصلاحيات قدر الإمكان
- سجلات المراجعة للعمليات الحساسة

### 4.3 Usability Requirements - متطلبات قابلية الاستخدام

**NFR-007: User Interface واجهة المستخدم**
- واجهة بديهية وسهلة الاستخدام
- تصميم responsive (موبايل، تابلت، سطح المكتب)
- دعم اللغتين العربية والإنجليزية

**NFR-008: User Experience تجربة المستخدم**
- تنقل واضح (navigation) ومعمارية معلومات (information architecture)
- رسائل أخطاء مفيدة
- مؤشرات التحميل للعمليات الطويلة (progress indicators)
- تصاميم متسقة

### 4.4 Reliability Requirements - متطلبات الموثوقية

**NFR-009: Data Integrity سلامة البيانات**
- اتساق المعاملات
- التحقق من صحة البيانات في جميع نقاط الإدخال
- إجراءات النسخ الاحتياطي والاسترداد
- تكرار البيانات

**NFR-010: Error hangling معالجة الأخطاء**
- معالجة سليمة للأخطاء
- رسائل خطأ سهلة الفهم
- تسجيل الأخطاء ومراقبتها
- إعادة المحاولة التلقائية لحالات الفشل

### 4.5 Compatibility Requiremets - متطلبات التوافق

**NFR-011: Browser Support دعم المتصفح**
- المتصفحات الحديثة (Chrome، Firefox، Safari، Edge)
- متصفحات الموبايل (iOS Safari، Chrome Mobile)

**NFR-012: Device Support دعم الأجهزة**
- أجهزة كمبيوتر سطح المكتب
- الأجهزة اللوحية
- الهواتف المحمولة
- تصميم متجاوب لجميع أحجام الشاشات

### 4.6 Maintainablity - متطلبات قابلية الصيانة

**NFR-013: Code Quality جودة الكود**
- كود موثق جيداً
- اتباع معايير البرمجة
- معمارية معيارية
- اختبارات الوحدة والتكامل (unit and integration tests)

**NFR-014: Documentation التوثيق**
- توثيق تقني
- أدلة المستخدم
- توثيق API
- أدلة النشر
